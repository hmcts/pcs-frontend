1. Executive Summary
The current Jenkins pipeline executes the pcs-frontend suite on Linux agents using bundled Playwright browser engines.
Key Finding: The PoC demonstrated significant behavioral divergence between native macOS/Windows environments and our Linux containers. These discrepancies—invisible to our current Jenkins setup—allow us to identify and mitigate platform-specific rendering and performance risks before they reach end users.
Final Recommendation: Adopt a Hybrid Model.
* Jenkins (Linux): Execute full regression suites (Fast, Cost-Effective).
* Sauce Labs (macOS/Windows): Run the matrix via yarn test:crossbrowser (delegates to yarn test:sauce:matrix → .sauce/config-matrix.yml). SAUCE_GREP is @crossbrowser in Jenkinsfile_nightly; package.json uses ${SAUCE_GREP:-@crossbrowser}. Optional single-suite entry points: yarn test:sauce:mac13-webkit (macOS 13 WebKit) and yarn test:sauce:* per platform — see package.json.

2. Strategic Justification: Why Sauce Labs?
As per TCOE guidance, Sauce Labs is reserved for cross-browser/OS validation, not general functional testing.
* WebKit Engine Divergence: Playwright on Linux uses a Linux-compiled WebKit engine. Sauce Labs runs WebKit on native macOS VMs (Safari-class stack). Behavior can differ from Linux WebKit in areas such as layout, visibility, and timing.
* OS-Specific Rendering: macOS and Windows handle CSS properties and stacking contexts differently than Ubuntu Linux.
* Risk Mitigation: Tests that pass on Linux but fail on macOS/Windows uncover "blind spots" in a standard Linux-only CI pipeline.

2.2 Can we use Microsoft Edge on Sauce?

Yes. This PoC already runs Edge on Windows 10 and 11 (Playwright project MicrosoftEdge; Chromium-based Edge). Sauce’s supported-browser column lists “Chromium” alongside Chrome—Edge is a Chromium-based desktop browser; Playwright drives it via the Microsoft Edge channel, not as a separate “Edge” engine name in that table. If a job fails, check Limitations (e.g. Chrome launchOptions overriding Sauce paths) rather than assuming Edge is unsupported.

2.3 Why Sauce Labs if we already use Playwright? (TCOE / stakeholder question)
Playwright is the test framework everywhere. Jenkins already runs Playwright on Linux with bundled Chromium, Firefox, and WebKit — fast and cheap for volume. That is not the same as:
* Safari-class WebKit on native macOS (Linux WebKit ≠ Safari stack behaviour).
* Google Chrome / Microsoft Edge on real Windows desktop OS images.
* Mobile Chrome on Android in the cloud (when the region/account supports Playwright Android — not demonstrated in this eu-central-1 POC).

Sauce Labs (or similar cloud browsers) is therefore for environmental parity on OS/browser combinations Linux CI cannot honestly represent — not because Playwright is missing features, but because the runtime environment differs. Cost and shared licences mean Sauce should stay on a narrow scope (e.g. nightly, happy-path / @crossbrowser-tagged subset), with full regression on Jenkins — see §6.1 (TCOE).

5. Key Findings: Environmental Divergence
The PoC identified discrepancies that passed on Jenkins but failed on Sauce Labs, highlighting the value of native platform testing:
❌ Discrepancy 1: Safari / WebKit Rendering (Address Selection)
* Observation: locator.selectOption: element is not visible.
* Technical Context: The element appeared usable on Linux WebKit but failed on macOS WebKit. Safari-class stacks treat visibility and stacking contexts more strictly.
* Impact: Surfaces a real-world blind spot for users on Apple devices.
❌ Discrepancy 2: Execution Latency ("Start now" button)
* Observation: Timeout (10000ms) exceeded.
* Technical Context: Native macOS/Windows runs can require more time for DOM resolution than local Linux agents.
* Impact: Highlights performance/timing sensitivity; underscores the need to tune timeouts for cross-platform suites.

6. Infrastructure and Resource Management (TCOE Alignment)
To align with the shared HMCTS organizational account:
* Concurrency: TCOE guidance suggests 3 parallel virtual jobs per product to avoid starving the ~25 shared licenses. Raising this (e.g., to 5) requires TCOE confirmation.
* Shared Pool Etiquette: Heavy Sauce matrices are scheduled for the Nightly Window to reduce contention with other teams.
* Stability: CI retries are recommended for transient 503 errors (test-composer) or license queuing delays.
* Artifacts: Full-journey screenshots and traces are enabled for Sauce; Allure is skipped. Video in the Sauce UI may show a placeholder; rely on Playwright Traces for evidence.

6.1 TCOE guidance (Satya) — pipeline strategy, limits, and best practice
The following aligns the POC with HMCTS TCOE input from the Sauce + Playwright CI/CD thread (Satya’s replies on pipeline, concurrency, limits, and happy-path scope). If Hemanth or others provided additional written constraints, add a dated bullet under this section so the POC record stays complete. It is guidance, not a legal quote — confirm with TCOE if policy changes.

Pipeline strategy
* Sauce is intended for the cross-browser phase, typically nightly — not necessarily every pipeline (preview/master/nightly all firing Sauce). Time-consuming; schedule to manage cost.
* Align Sauce frequency with how often UI changes: use Sauce to catch browser/OS-specific issues when UI changes warrant it — not every trivial change by default.

Parallel execution / concurrency (org limits)
* Organisational pool (shared across teams): order-of-magnitude — ~25 parallel virtual cross-browser sessions, 1 real device, 500 visual snapshots (TCOE; verify current numbers with TCOE).
* Per product: prefer not more than ~3 parallel threads for your tests; run at times that do not starve other teams.

Usage limits and behaviour
* Real-device tests are more constrained than virtual browsers; virtual sessions may wait until a licence is available.
* Most HMCTS teams use WebdriverIO; Playwright + saucectl can behave differently — validate limits and docs for Playwright specifically (this POC uses saucectl + Playwright).

Best practice (fast feedback vs cost)
* Prefer happy-path / critical UI scenarios on Sauce for cross-browser runs — not every edge case (edge cases stay on Jenkins/Linux where practical).
* Balance: fast feedback on PR/dev via Jenkins; Sauce for targeted cross-browser / nightly as agreed with TCOE.

7. Proposed Operating Model
1. Fast Feedback: Continue running the bulk of tests on Jenkins (Linux) for immediate developer feedback.
2. Cross-Browser Sanity (Nightly): Trigger Sauce for the matrix / single-suite jobs (SAUCE_GREP @crossbrowser on Sauce). Jenkins Playwright E2E remains @nightly (PLAYWRIGHT_TEST_GREP)—do not conflate the two tags when comparing timings.
3. Resource Management: Virtual browsers wait in a queue when licenses are saturated.
4. Mobile Strategy: Playwright Android/iOS in EU is currently out of scope; real-device strategy remains a separate decision.

8. Final Verdict: Hybrid Model
The PoC showed that native macOS WebKit is not identical to Linux WebKit, and that Windows desktop Chrome/Edge add further parity. While Jenkins remains the engine for speed and volume, Sauce Labs is appropriate for final environmental validation on stacks that Linux agents do not represent.
A nightly hybrid model preserves environment parity without inflating day-to-day CI cost, adhering to agreed TCOE concurrency and scheduling limits.
